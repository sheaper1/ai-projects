## 1. TL;DR

Конвейер: эталонная библиотека Gutenberg-блоков → из неё собираются сайты под
клиентов. Активный проект — `projects/rosenberger/` (блочная тема + плагин).

**Перед работой:** `git pull` → прочитай хвост `CHANGELOG.md`.
**Делая блок:** бери ТОЧНЫЕ значения из Figma (не на глаз) → стили только на токенах
`theme.json` → ширины в **px** (контейнер 1344px по центру, `vw` только в Hero) →
повторяющийся контент = **репитер** (add/remove/↑↓) → медиа через `MediaUpload` с
превью над кнопкой.
**Закончив:** `npm run build` → `php -l` → `node scripts/deploy-stack.mjs` →
проверь в браузере → строка в `CHANGELOG.md` → коммит `[Claude] …` → `git push`.
**Для ровного качества (и на слабой модели тоже):** иди по чек-листу §6a
(reuse-first → значения из Figma → ассеты webp/svg → сверка каждой секции
`npm run shot`). Это главный раздел про «как не терять точность».
**Не нарушать:** дизайн один-в-один по Figma, без отсебятины. Не коммить `.env`,
`build/`, `node_modules/`, `.claude/`, `.figma-tmp/`.

## 2. Что это за проект

Веб-агентство собирает клиентам сайты на WordPress/Gutenberg из готовых
динамических блоков. Клиент сам редактирует контент (тексты, картинки), а **макет
защищён** — структуру он не ломает.

- **`/library/`** — ЭТАЛОННАЯ библиотека блоков (read-only-ориентир). Новый блок
  сначала доводится здесь, потом копируется в проект.
- **`/projects/<сайт>/`** — конкретный сайт = самодостаточная **блочная тема**
  (`theme/`) + **плагин проекта** (`plugin/<name>-core/`).
  - Тема (ВИД): `theme.json`, шаблоны, `blocks/<slug>/`. Тема сама регистрирует
    блоки из `functions.php` (glob по `blocks/*`).
  - Плагин (ДАННЫЕ/ЛОГИКА): настройки сайта, CPT, Block Bindings, формы — чтобы
    пережили смену темы.
- **Модель «копия в проект»**: блок копируется из `/library` в тему и дальше
  правится **только копия в проекте**. Эталон не трогаем (улучшения эталона —
  будущим проектам).

Активный проект: **`projects/rosenberger/`** (немецкий риелтор, дизайн в Figma
`p1HKLfoMcOwtVUD5rI9V3P`, страница «UI Design» `142:3`).

## 3. Карта репозитория

```
/AGENTS.md                  ← этот файл (мастер-инструкция)
/CLAUDE.md                  ← архитектура и «почему»
/CHANGELOG.md               ← журнал: что сделано, дописывать перед push
/theme.json                 ← базовый эталон токенов (шаблон для проектов)
/package.json               ← npm-скрипты (build / build:library)
/scripts/
  build-blocks.mjs          ← сборка блоков каталога через wp-scripts
  deploy-stack.mjs          ← деплой темы+плагина на staging + аплоад медиа
  scaffold-cpt.mjs          ← генерация PHP-файлов для нового CPT (см. §7)
  cpt-templates/example.json ← шаблон конфига для scaffold-cpt
/library/
  blocks-registry.md        ← каталог всех блоков (СВЕРЯТЬСЯ перед созданием)
  blocks/<slug>/            ← эталон блока
/projects/rosenberger/
  theme/
    theme.json              ← ДИЗАЙН-ТОКЕНЫ ПРОЕКТА (единственный источник)
    functions.php           ← регистрация блоков (glob blocks/*)
    blocks/<slug>/          ← блоки проекта (правим здесь)
    parts/, templates/      ← шапка/подвал/шаблоны
    style.css               ← заголовок темы
  plugin/rosenberger-core/
    rosenberger-core.php    ← подключает includes/*
    includes/settings.php   ← таб «Настройки сайта»
    includes/bindings.php   ← Block Bindings (rosenberger/setting)
    includes/cpt.php        ← кастомные типы записей
    includes/svg-media.php  ← безопасная загрузка SVG для админов
  media/                    ← исходные ассеты (иконки, фото) для аплоада на staging
.env                        ← доступы WP (gitignore): WP_URL, WP_USER, WP_APP_PASSWORD
```

Структура одного блока:
```
blocks/<slug>/
  block.json        ← apiVersion 3; editorScript/style/render; viewScript если нужен фронт-JS
  render.php        ← серверный рендер: обёртка-секция + контент
  view.js           ← (опц.) ФРОНТ-JS как ПЛОСКИЙ файл в корне блока (не в src/!)
  src/
    index.js        ← registerBlockType(metadata.name, { edit, save: () => null })
    edit.js         ← редактор: InspectorControls + превью канваса
    style.scss      ← фронт + редактор, ТОЛЬКО токены
  build/            ← генерится npm run build; в git НЕ коммитим (.gitignore)
```

## 4. Золотые правила

Эти правила выстраданы фидбеком пользователя. Нарушение = переделка.

1. **Дизайн один-в-один по Figma. Без отсебятины, но не слепо.** Цвета, размеры, отступы,
   ассеты бери ПРОГРАММНО из Figma (`get_design_context`, `get_variable_defs`,
   `get_screenshot`), а не подбирай на глаз. Если в макете `#f8f5f3` — значит он
   (через токен). Картинки из макета подставляй (не оставляй пустые боксы). Если
   данные живые и контент может быть длиннее макета, сохраняй визуальный принцип
   Figma, но не ломай адаптив и не фиксируй высоты там, где секция должна
   естественно расти по самому высокому элементу ряда.
2. **Стандарт качества держится в каждой сессии.** Обязан держать один
   и тот же уровень визуальной и технической точности: сначала точный frame/node
   из Figma, потом правка; сначала минимальное решение, потом проверка в браузере;
   не оставлять "почти похоже"; не принимать системный UI, кривые мобильные
   высоты, случайные отступы и разъехавшийся ритм карточек как норму.
3. **Рабочий цикл для визуальных задач обязателен.** Для блоков и секций порядок
   такой: 1) сверка с Figma, 2) точечная правка в проектной копии блока, 3)
   синхронизация в `library`, 4) `npm run build`, 5) `node scripts/deploy-stack.mjs`,
   6) повторная визуальная проверка staging. Если после деплоя видно расхождение,
   задача не считается законченной.
4. **Живой контент важнее слепой фиксированной геометрии.** Если карточки,
   списки, фильтры, заголовки или подписи приходят из данных, высота и
   выравнивание должны строиться от контента и сетки, а не от жёсткой цифры,
   если только эта цифра не действительно обязательна по макету и не ломает
   адаптив.
5. **Везде только UTF-8.** Все новые и редактируемые файлы, строки интерфейса,
   комментарии, JSON, PHP, JS и служебные тексты сохранять в UTF-8 без битой
   кириллицы и немецких символов. Любые `Р...`, `Г...`, `вЂ¦` и похожие артефакты
   считаются багом и должны исправляться до коммита.
   **ПРИЧИНА мохибейка (был в `CHANGELOG.md`):** запись/дозапись текстовых файлов
   через PowerShell (`Out-File`/`Set-Content`/`>>`/`Add-Content`) — у PS дефолт
   UTF-16/локальная кодовая страница → кириллица превращается в `Р·Р°...`.
   **ПРАВИЛО:** русские/немецкие тексты (`CHANGELOG.md`, render.php, scss…) писать
   и дозаписывать ТОЛЬКО инструментами Write/Edit (они пишут UTF-8), не через
   PowerShell-редирект. Чинить мохибейк детерминированно: `iconv.encode(str,
   'win1251').toString('utf8')` (есть `iconv-lite` в deps).
6. **Изображения и иконки — только из Figma.** Для макетных фото, иконок, SVG,
   фоновых изображений и логотипов источник один: Figma-ассеты/экспорт из Figma.
   Нельзя вырезать изображения из скриншотов, подбирать похожие вручную или
   временно подменять их не-Figma файлами, если пользователь явно не разрешил.
7. **Стили только через токены `theme.json`.** `var(--wp--preset--…)`,
   `var(--wp--custom--…)`. Захардкоженные цвета/шрифты/размеры запрещены — даже в
   редакторских контролах. Нужен новый повторяемый параметр → сначала токен в
   `theme.json`, потом используй.
8. **Ширины в пикселях.** Контент — в центрированном контейнере
   `max-width: 1344px; margin: 0 auto` (это `wideSize`, поля по 48px при 1440).
   Секцию НЕ растягивать на весь экран. `vw`/`vh` использовать **только в Hero**
   (первый экран — ВСЕГДА `vw` и `vh`: высота на `vh`, координаты/размеры контента на
   `vw`/процентах; mobile — отдельный `100svh`). Колонки — px / `fr` / `%`, но не `vw`.
   Структурная разметка: фон и боковые/верт. отступы — на самой секции; контент —
   во вложенном контейнере `__inner`.
9. **Повторяющийся контент = репитер.** Списки/карточки/преимущества делать с
   управлением строками (добавить / удалить / ↑↓) по образцу `hero-cover`
   (`addItem/removeItem/moveItem` над атрибутом-массивом). render.php бежит
   `foreach` — серверную часть и CSS при добавлении репитера менять не нужно.
10. **Медиа — через `MediaUpload` с превью.** Превью текущего фото/иконки —
   **крупное и НАД кнопкой** выбора (так сразу видно, что выбрано). Клик по превью
   тоже открывает медиатеку. Иконки/фото хранятся в медиатеке (media ID/URL);
   inline SVG и icon-font запрещены (исключение — служебные ассеты темы).
11. **Только динамические блоки** (`render.php`), без статического HTML в `save`
   (`save: () => null` или `<InnerBlocks.Content />`).
12. **Макет защищён.** Клиент правит текст/картинки, но не ломает структуру.
13. **Адаптив — ПО МОБИЛЬНОМУ МАКЕТУ, не на глаз.** В Figma есть отдельная колонка
   `home mobile` (`2005:2338`, 375px) со своим фреймом для каждой секции. Бери
   мобильную раскладку, размеры, отступы ИЗ НЕЁ, а не выдумывай брейкпоинты.
   Мобильная ширина 375, поля 16px (контент 343px). Соответствия:
   Hero `2005:2339`, trust `2005:2359`, pain points `2005:2377`, cards `2005:2443`,
   about — Frame 21 mobile. Проверяй планшет (≤1024) и мобайл (≤782/375) в браузере.
14. **Library-first + переносимые глобальные стили.** Новый переиспользуемый блок
   сначала полностью доводи в `/library`, затем копируй в проект; в проектной копии
   меняй только клиентские отличия. Общие состояния компонентов (button
   hover/focus, `box-sizing`, защита от mobile overflow) задавай один раз в
   глобальных стилях темы на токенах. Для `width:100%` у кнопок обязателен
   `box-sizing:border-box; max-width:100%`. Существующий проектный форк не
   синхронизируй обратно в library автоматически: переноси только доказанное
   универсальное улучшение. Когда правишь баг в проектной копии блока, спроси
   себя — это баг эталона? Если да, прогони `npm run diff:blocks -- --patch <slug>`,
   посмотри построчно «фикс vs клиентская адаптация» и перенеси в `/library`
   только универсальное. Так фиксы не теряются при росте числа проектов.
15. **Следуй Figma грамотно.** Для каждой секции связывай desktop-node с её
   соответствующим mobile-node и снимай оба до кода. Точные визуальные параметры
   повторяй, но не переноси в живой сайт ограничения макета, которые ломают
   динамический контент: фиксированную высоту заменяй `min-height`/естественной
   высотой, сохраняя композицию, отступы и пропорции.
16. **Переносы строк — ровно как в Figma.** В заголовках/подписях переносы сделаны
   осознанно (напр. «Sie verkaufen, / ich mache den Rest», «Häufige Fragen / zum
   Verkauf»). Не полагайся на автоперенос: бери `<br aria-hidden />` из
   `get_design_context` и ставь `<br>` ровно там же (рендер — `wp_kses_post`, не
   `esc_html`). Сверяй переносы на скриншоте staging против макета.

Две модели редактирования (обе валидны):
- **Атрибуты + репитер** (как `hero-cover`, `cards-stack`, `pain-points`,
  `trust-bar`, `about`) — для контентных секций со своей точной разметкой/CSS.
  Это основной путь в проекте.
- **InnerBlocks + `templateLock:"all"`** (как `library/hero`) — клиент правит
  вложенные ядровые блоки инлайн; `render.php` = обёртка + `$content`,
  `save = <InnerBlocks.Content />`.

## 5. Дизайн-токены

Источник — `projects/rosenberger/theme/theme.json`. Контейнеры: `contentSize`
864px (текст), `wideSize` 1344px (секции/сетки).

**Цвета** (`--wp--preset--color--<slug>`):

| slug | hex | назначение |
|------|-----|-----------|
| `base` | `#ffffff` | белый (карточки, светлый фон) |
| `contrast` | `#142335` | тёмно-синий (текст, тёмный фон, кнопки) |
| `primary` | `#142335` | акцент бренда |
| `accent` | `#decec4` | бежевый (иконки-подложки) |
| `surface` | `#f8f5f3` | тёплый светлый фон секций |
| `surface-muted` | `#f1f2f3` | вторичная подложка |
| `muted` | `#747c86` | приглушённый текст (описания) |
| `border` | `#b2b3b7` | границы |
| `placeholder` | `#969696` | плейсхолдеры |
| `on-dark` | `#ffffff` | текст поверх тёмного фона |

**Шрифты** (`--wp--preset--font-family--<slug>`): `heading` = Nunito Sans,
`body` = Roboto Flex.

**Размеры текста** (`--wp--preset--font-size--r-*`): `r-12 r-14 r-16 r-18 r-20
r-24 r-32 r-40 r-48 r-56 r-64 r-80`. Префикс `r-` ОБЯЗАТЕЛЕН (стандартные slug'и
Gutenberg может переопределить сохранённый в БД Global Styles).

**Интервалы** (`--wp--preset--spacing--r-*`): `r-4 r-8 r-16 r-24 r-32 r-40 r-48
r-64 r-section`.

**Кастом** (`--wp--custom--…`):
- `layout--gutter` = `clamp(1.25rem, 3.333vw, 3rem)` — боковой отступ секции.
- `layout--section-space` = `var(--wp--preset--spacing--r-section)` — верт. отступ.
- `radius--small|medium|large|pill` = `0.5rem | 1rem | 1.5rem | 999px`.
- `transition--fast|base` = `160ms ease | 240ms ease`.

> Не создавай в блоке свою палитру/шкалу/контейнер. Нужен новый повторяемый
> параметр — сначала токен в `theme.json`.

**Маппинг «значение из Figma → токен» (обязательный шаг, экономит токены ИИ).**
Figma-фрейм почти НЕ использует переменные — отдаёт сырые `#hex`/`px`. Не угадывай
имя токена и НЕ перечитывай `theme.json` в контекст — спроси скрипт:

```
npm run tokens -- "#f8f5f3" 24px 80px 724px   # → var(--wp--preset--…); ✗ если токена нет
npm run tokens:scan                            # линт SCSS блоков на хардкод-цвета
```

- `tokens <значения…>` — детерминированный резолвер (только парс `theme.json`):
  отдаёт имя токена или `✗ токена нет` (тогда сначала заведи пресет в `theme.json`).
- `tokens:scan` — ловит `#hex`-литералы в `blocks/*/src/*.scss`; `✗` = есть токен,
  правь обязательно; `⚠` = токена нет (alpha/градиент допустимы). **Встроен в
  `npm run check` и pre-push** — `✗` блокирует push. Fallback'и вида
  `var(--токен, #hex)` тоже считаются хардкодом: в этой теме токены всегда
  определены, пиши чистый `var(--токен)`.

## 6. Как собрать новый блок

1. **Сверься с тем, что уже есть — ОБЯЗАТЕЛЬНО, до создания.** Запусти
   `npm run blocks` (печатает каталог из `block.json`: slug/категория/описание/
   keywords — всегда актуально). Похож на нужный → **копируй и допили, не лепи с
   нуля**; нет → создавай новый и в конце **допиши строку в `blocks-registry.md`**.
2. **Сними дизайн из Figma**: `get_design_context` для node (даёт код + ассеты +
   скриншот). Зафиксируй точные размеры, цвета, тексты, ссылки на ассеты. Сырые
   `#hex`/`px` прогоняй через `npm run tokens -- <значения…>` — он отдаст имя
   токена (не угадывай и не перечитывай `theme.json`). Если `✗ токена нет` —
   сначала заведи пресет в `theme.json`. При огромном ответе метаданных — читай
   через python из сохранённого tool-result, а не целиком в контекст.
3. **Скачай ассеты** из макета (`curl` по URL). Тяжёлые PNG/фото → конвертируй в
   WebP (`sharp`, ресайз ~1920px, q80) и положи в `projects/rosenberger/media/`.
4. **Создай блок в `/library/blocks/<slug>/`** (block.json, render.php, src/).
   Соблюдай золотые правила (токены, px, контейнер `__inner`, репитер, превью).
5. **Скопируй блок в тему** `projects/rosenberger/theme/blocks/<slug>/`. Дальше
   правишь обе копии синхронно (или только тему, если эталон уже зафиксирован).

## 6a. Playbook качества (чек-лист на каждую секцию)

> Эти приёмы дали ровную точность на странице immobilie-verkaufen. Делай по ним
> механически — тогда и слабая модель (Sonnet) держит тот же уровень. Не полагайся
> на «глаз» и память: каждый шаг — конкретная команда/инструмент.

**Ритм одной секции (повторяй для каждой):**
1. **Reuse-first.** `npm run blocks` → сопоставь секцию Figma с существующим блоком
   по `block.json`-атрибутам. Похож — переиспользуй (конфиг в import-скрипте);
   раскладка реально другая — новый блок. НЕ лепи новый, если подходит старый.
2. **Точные значения из Figma — программно.** `get_design_context(node)` (код +
   ассеты + скрин), `get_metadata` (структура; ОГРОМНЫЙ ответ — сохрани в файл и
   парси python/jq, НЕ тяни в контекст), `get_variable_defs`, `get_screenshot`.
   Не подбирай на глаз. Сырые `#hex`/`px` → `npm run tokens -- <значения>`.
3. **Ассеты.** Фото → WebP (`sharp`, ~1920px, q80) в `media/`. Иконки → SVG через
   `download_assets(format=svg)` → **`npm run figma:icon -- <файл.svg>`** (убирает
   фон-холст `#1E1E1E` и геометрию страницы из экспорта) → медиатека. Иконки НЕ PNG.
4. **Переносы строк** — `<br>` ровно как в макете (правило §16).
5. Собери/обнови блок и import-скрипт → `npm run build` → `npm run check`
   → `node scripts/deploy-stack.mjs` → `node scripts/import-<страница>.mjs`.
6. **Визуальная сверка КАЖДОЙ секции (обязательна, не пропускать):**
   `npm run shot -- <url> --sel ".wp-block-library-<slug>" --name <slug>` →
   открой PNG (Read) → сравни с нодой Figma (`get_screenshot`) → чини, пока не
   совпадёт. Мобайл: `npm run shot -- <url> --mobile --full` (сам нарежет).
   Тул headless (puppeteer), Chrome-расширение НЕ нужно — «браузер недоступен»
   не отговорка (см. §11). Нет гейта → нет «готово».
7. Строка в `CHANGELOG.md` → маленький коммит `[Claude] …` → `git push`.

**Хитрости, которые экономят переделки:**
- **Бренд/сторонние цвета** (золото звёзд, Google-логотип) пиши **инлайном в
  `render.php`**, не в SCSS: `tokens:scan` сканит только SCSS, а в макете эти
  цвета — часть ассета (как логотип). Токены остаются чисты.
- **Карусели — ТОЛЬКО через общий движок `theme/assets/js/rb-carousel.js`
  (`window.RbCarousel`).** Любая карусель ОБЯЗАНА иметь: drag мышью + свайп тачем
  (pointer events) и **бесконечный цикл** (клоны набора слева/справа, бесшовный
  rebase) — статичный «листатель без петли и драга» не принимается. Разметка:
  контейнер с `[data-track]` (дети = слайды), опц. `[data-prev]/[data-next]/
  [data-dots]`. Вызов из `view.js` блока: `RbCarousel(el, { frame, clickToGo,
  autoplay })`. Скрипт подключён глобально в `functions.php`. Образцы: `property-hero`
  (full-width, 1 слайд), `property-gallery` (coverflow: `frame` = viewport,
  `clickToGo:true`, активный по центру через `.is-active`).
- **Анимации — CSS, не JS-высота.** Аккордеон = `grid-template-rows: 0fr↔1fr`
  (плавно, без рывков); padding выноси во вложенный элемент, иначе `0fr` не
  схлопнется в 0. JS только переключает класс. Прогрессивное улучшение: без JS
  контент открыт.
- **Данные стороннего плагина** (напр. Google-отзывы grw): не парси HTML — найди
  хранилище (опции/transient) разовым Code-Snippet с временным REST-роутом, потом
  helper в `plugin/<core>/includes/` и читай из блоков. PHP-массивы в сниппете —
  через `json_decode('<json>', true)`, не JS-литерал `[{…}]` (это парс-ошибка PHP).
- **«Не как в дизайне».** Сними ИМЕННО эту ноду (`get_screenshot(nodeId)`),
  сравни обводки/размеры/цвет и повтори; не доверяй догадке.
- **Среда.** bash `/tmp` ≠ Windows-node: temp-файлы качай внутрь проекта.
  Скриншоты >2000px по стороне просмотрщик отклоняет — `npm run shot` сам ужимает.
- **Один источник правды по странице** — `scripts/import-<страница>.mjs`:
  переналив контента дёшев, правь там, а не в БД руками.
- **Фильтры и сортировки — ВСЕГДА AJAX, без перезагрузки страницы.** Любой
  каталог/список с фильтром или сортировкой: REST-эндпоинт в плагине отдаёт
  готовый HTML сетки (одна общая PHP-функция рендера — без дубля шаблона в JS),
  `view.js` подменяет область результатов и обновляет URL через
  `history.replaceState` (ссылки остаются шарабельными). Обычная GET-перезагрузка
  формы — только как no-JS фолбэк. Образец: `property-catalog` +
  `plugin/.../includes/property-catalog.php` (`rosenberger/v1/properties`).

**Эскалация в `HANDOFF-OPUS.md` — по ФАКТУ-гейту, не по самооценке.**
Не предсказывай заранее «смогу / не смогу» (модель оценивает себя плохо) — **пробуй,
потом решает гейт**. Эскалируй секцию, если сработал любой триггер:
- **Структурный блокер:** секция требует того, что §6a-цикл не даёт за 2 захода —
  интерактивный блок с нетривиальной JS-логикой (многошаговая форма, калькулятор),
  новый тип блока вне библиотеки, серверная логика (`wp_mail`, связки с CPT). Не
  городи костыль — эскалируй сразу.
- **Гейт качества не сходится:** после 2 честных заходов визуальная сверка §6a п.6
  (`shot` ↔ Figma; при сомнении `npm run visual`) в зоне секции упорно расходится.
  Не пушь кривое и не точи бесконечно.

При эскалации: **не блокируй всю страницу** — поставь временную заглушку (ближайший
подходящий блок, напр. `consultation-cta`) или пропусти секцию и доведи остальные.
Затем допиши **структурную** запись в `HANDOFF-OPUS.md` по формату файла (страница,
место/секция, Figma-нода, что нужно, что мешает — с числом расхождения, если это
гейт, — шаги «после готового блока»), отметь строкой в `CHANGELOG.md` и коммитом.
Цель: Sonnet добивает максимум сам (это приоритет), к Опусу уходит только
доказанно непосильное — **по измерению, а не по предчувствию**.

## 7. CPT — скаффолдинг нового типа записей

**Принцип:** CPT/таксономии/мета — всегда в **плагине** (`plugin/<name>-core/includes/`), не в теме. Так данные переживают смену темы.

### Быстрый старт

```bash
# 1. Создай конфиг по шаблону
cp scripts/cpt-templates/example.json scripts/cpt-templates/<slug>.json
# 2. Отредактируй конфиг (slug, labels, taxonomies, metaFields)
# 3. Сгенерируй PHP-файлы
node scripts/scaffold-cpt.mjs scripts/cpt-templates/<slug>.json
# 4. Следуй чеклисту в выводе скрипта
```

Флаг `--dry-run` — показать сгенерированный PHP в консоль без записи.

### Что генерирует скрипт

| Файл | Содержимое |
|------|-----------|
| `includes/<slug>-cpt.php` | `register_post_type` + `register_taxonomy` (N штук) + `register_post_meta` |
| `includes/<slug>-meta-box.php` | meta box с 2-column grid, text/select-поля, nonce + save hook |

### Что сделать вручную после генерации

1. **Подключить в плагине** — добавить 2 строки `require_once` в `<plugin>-core.php`
2. **Сбросить перемалинки** — WP Admin → Settings → Permalinks → Save (один раз после деплоя)
3. **Блок отображения мета** — `theme/blocks/<slug>-meta/` (render.php, block.json). Пример — `property-meta`. Использует `$block->context['postId']` для query-loop-совместимости.
4. **Шаблоны** — `templates/single-<slug>.html` и `templates/archive-<slug>.html`. Структура — по аналогии с `single-property.html` / `archive-property.html`.
5. **CSS карточки** — в `theme/style.css`
6. **Seed-скрипт** — по аналогии с `scripts/seed-properties.mjs`; мета пишем через Code Snippets если ACF блокирует REST

### Грабли CPT

- **`show_in_rest: true`** нужно и для CPT, и для каждой таксономии, и для каждого мета-поля — иначе Gutenberg не видит данные.
- **`'default' => ''`** в `register_post_meta` — REST API вернёт `""` вместо `null` для незаполненных полей; render.php проверяет `empty()` и работает в обоих случаях, но явный default чище.
- **ACF блокирует REST meta** — на staging установлен ACF, который убирает ключ `meta` из REST-ответа. Читать мета через PHP (`get_post_meta`) в render.php — работает. Писать через REST — нет. Обходной путь: Code Snippets (см. `seed-properties.mjs` → `writeMeta()`).
- **404 на архиве** после первого деплоя — нужен сброс перемалинок (п. 2 выше).
- **`save_post_<slug>`** хук — при использовании `save_post` (без суффикса) hook срабатывает на все типы; лучше `save_post_<slug>` как в шаблоне.

## 8. Цикл завершения

Для любой задачи, меняющей код/тему/плагин/контент — **без напоминаний**:

1. **Проверка**: `npm run build` (без ошибок) + `npm run check` (php -l изменённых
   PHP + запрет секретов/артефактов + `tokens:scan` хардкодов + `lint:blocks`
   структурный линтер: динамический save, import style.scss, валидный block.json,
   варнинги по inline-svg/png/каруселям). Падает `check` → push заблокирован.
2. **Деплой**: `node scripts/deploy-stack.mjs`; убедись, что тема и плагин активны
   и staging отвечает (HTTP 200).
2a. **Визуальная сверка с Figma (правки вида) — ОБЯЗАТЕЛЬНО**, не на глаз:
   - сними полноширинный PNG фрейма (см. §10, трюк с `maxDimension`);
   - `npm run visual -- <url> --width 1440 --figma .visual/figma-<name>-1440.png --out .visual/<name>`
     и то же с `--width 375` для мобильного фрейма;
   - открой `*-side.png` (макет | реализация) и `*-diff.png` (heatmap); по числам
     зон найди, ГДЕ разъехалось, и доведи. Высокая дельта высот / красные зоны в
     середине = смещён ритм секций. Это советник, не push-гейт: живой контент и
     рендер шрифтов дают фоновый шум — сверяй артефакты глазами, чини структурное.
3. **Строка в `CHANGELOG.md`** (формат — в шапке файла), новые записи сверху.
4. **Коммит** только связанных файлов: `[Claude] …`, кратко по-русски.
   НЕ коммить `.env`, `build/`, `node_modules/`, `.claude/`, `.figma-tmp/`, чужое.
5. **`git push`** и проверь, что удалёнка приняла коммит.
6. **Всегда дай пользователю ссылку** на затронутую страницу в финальном ответе:
   `<staging>/<slug>/` (для контента — ещё ссылку на редактор
   `/wp-admin/post.php?post=<id>&action=edit`). Не заставляй искать руками.

Не прошло — НЕ выдавай за готовое: чини или явно назови блокер.
(Чисто аналитические задачи без правок кода — деплой/коммит не требуют.)

## 9. Сборка и деплой

```
npm install
npm run build           # собрать блоки проекта (инкрементально: только изменённые)
npm run build -- --force        # полная пересборка всех блоков
npm run build:library   # собрать эталоны (для проверки)
npm run diff:blocks     # сверить дрейф library ↔ theme (что разошлось)
npm run diff:blocks -- --patch <slug>   # построчный дифф блока (решить: фикс в эталон vs клиентская адаптация)
npm run page -- <spec>          # генератор страниц из спеки scripts/pages/<spec>.mjs (--dry = собрать офлайн без записи)
npm run shot -- <url> --sel ".wp-block-library-<slug>" --name <slug>  # скрин секции (моб.: --mobile --full)
npm run figma:icon -- <файл.svg>          # почистить SVG-экспорт иконки из Figma
npm run check           # предполётные проверки (php -l + секреты/артефакты + tokens:scan + lint:blocks)
npm run lint:blocks     # структурный линтер блоков (save/scss-import/block.json + варнинги)
# /review-block <блок|страница>  # критик block-reviewer по REVIEW.md + Figma (судейские правила)
node scripts/deploy-stack.mjs            # полный деплой темы + плагина на сервер
node scripts/deploy-stack.mjs --changed  # залить только изменённые файлы (диф-кэш)
```

- `npm run build` зовёт `wp-scripts build` на изменённые `blocks/<slug>/src`
  (пропускает блоки, чей `build/` свежее `src/`; полная пересборка — `-- --force`).
- `build/` в git НЕ хранится → перед деплоем нужен `npm run build`.
- `--changed` у деплоя сверяется с локальным `.deploy-cache.json` (gitignore,
  per-machine): нет кэша → полная заливка; первый деплой на машине делай полным.
- **Деплой** (`deploy-stack.mjs`): заливает файлы темы+плагина и активирует через
  одноразовый сниппет Code Snippets, который сам себя обезвреживает. Доступы — в
  `.env`. Также грузит медиа в медиатеку (`ensureMedia(slug, file, ext)`).
- ⚠️ В этой версии Code Snippets REST DELETE сломан (500) — служебные сниппеты не
  удаляются, а **обезвреживаются** (деактивация + пустой код). Это норма.
- Медиа в медиатеке идемпотентны: грузятся по `slug`, повторный деплой не дублирует.

## 11. Грабли (важно!)

Реальные ошибки, уже наступленные — не повторять:

- **`view.js` не в `src/`.** wp-scripts собирает `view.js` только если `block.json`
  лежит ВНУТРИ `src/`, а у нас он уровнем выше. Решение: фронт-JS — плоский файл
  `view.js` в корне блока + `"viewScript": "file:./view.js"`. Пиши чистый DOM-JS
  без импортов (его не бандлят).
- **Тёмный заголовок на тёмной секции.** Глобальные стили темы красят `h2/h3` в
  `contrast` (тёмный) и перебивают наследование `color` секции. На тёмном фоне
  задавай заголовку явный `color: var(--wp--preset--color--on-dark)`.
- **Секция растягивается на весь экран.** Нет контейнера `__inner` с `max-width`.
  Оборачивай контент в `.<block>__inner { max-width:1344px; margin:0 auto }`.
- **Фон обрезает важную часть.** `object-fit:cover` режет по центру. Подбирай
  `object-position` (напр. `right top`, чтобы не срезать голову на портрете).
- **Тяжёлый PNG из Figma.** Портреты/фото бывают по 5+ МБ. Конвертируй в WebP
  через `sharp` (resize ~1920, q80) → десятки КБ.
- **Sticky/transform в редакторе мешают правке.** Глуши их под
  `.editor-styles-wrapper` (`position:static; transform:none`).
- **`get_metadata` огромный** (вся страница > лимита токенов). Сохраняется в
  tool-result файл — извлекай нужное через python/jq, не тяни целиком в контекст.
- **Не теряй чужую работу.** Перед стартом `git pull`; если в дереве чужие
  незакоммиченные изменения — разберись, чьи, и не перетирай.
- **Превью медиа маленькое/сбоку плохо видно.** Делай крупным и НАД кнопкой.
- **Новые страницы — скрипт, НЕ HTML-шаблон.** `templates/*.html` существуют
  только для типов записей: `page.html`, `single-*.html`, `archive-*.html`. Для
  конкретной страницы (Über mich, Kontakt и т.п.) создавай
  `scripts/import-<slug>.mjs` по образцу `import-homepage.mjs` — создаёт/обновляет
  страницу через WP REST API с блочной разметкой в поле `content`. Файл
  `templates/page-<slug>.html` — это ловушка: контент не хранится в файле, он
  в базе WP; пустой шаблон ничего не выведет.
- **`single-*` / `archive-*` (детальная/каталог записи CPT) — это ШАБЛОН, не
  страница.** НЕ создавай импорт-скрипт и НЕ «собирай страницу». Контент
  динамический — из самой записи: правь `templates/single-<cpt>.html` /
  `archive-<cpt>.html` динамическими блоками (`post-title`, `post-featured-image`,
  `post-content`, кастомные блоки вроде `rosenberger/property-meta`, читающие
  текущую запись). Дизайн-матч ведёшь по Figma-ноде сингла/архива, а **проверяешь
  на реальном URL записи** (`/objekte/<slug>/`), не на созданной странице. Шаг §6a
  «`import-<страница>.mjs`» здесь НЕ применяется.
- **Скриншот staging = `npm run shot` / `npm run visual` (headless puppeteer).**
  Им НЕ нужен Chrome-расширение, computer-use или браузерный MCP — они снимают сами.
  «Расширение/браузер недоступен» — **НЕ повод пропускать визуальный гейт §6a п.6**:
  бери `npm run shot -- <url> --sel ".wp-block-library-<slug>" --name <slug>`, читай
  PNG (Read), сравни с Figma. НЕ трать заходы на Chrome MCP / computer-use ради
  снимка staging — это тупик (read-only/не подключён), а puppeteer-тул работает.
- **REST отдаёт 500 при сохранении страницы, но контент СОХРАНЯЕТСЯ.** Известный
  сайд-эффект хука кеша на staging. НЕ диагностируй PHP и НЕ глуши ошибку в
  импорт-скрипте — просто проверь результат: `GET /<slug>/` → 200 и
  `/wp-json/wp/v2/pages/<id>?context=edit` → контент на месте. Не уходи в отладку
  фатала, которого нет.

## 12. Экономия токенов

Пользователь следит за расходом — дёшево и точно:

- **Не перечитывай** файлы без нужды; читай нужный фрагмент (offset/limit), для
  поиска — Grep/Glob, а не чтение всего подряд.
- **Параллель**: независимые чтения/команды — одним батчем.
- **Не пересказывай** известное и не описывай варианты, которые не делаешь. Решил —
  делай, без обзора.
- **Маленькие коммиты** по одной задаче.
- **Surgical changes**: трогай только нужное, чужой стиль не «улучшай».
- Огромные tool-result (Figma-метаданные) — обрабатывай скриптом, не контекстом.
- **Маппинг значений в токены — через `npm run tokens`**, а не перечитыванием
  `theme.json` в контекст (см. §5). Это прямая экономия токенов ИИ.
- **Визуальный дифф:** числа по зонам в stdout — дёшевы, читай их сначала.
  Тяжёлые `*-side.png`/`*-diff.png` открывай в контекст ТОЛЬКО для зон с высоким
  %, а не подряд (картинки дороги). Можно сделать thumbnail через `sharp` и
  смотреть его.

## 13. Git

- **Старт работы → `git pull`. Конец → `git push`.** Забыл push — следующая
  сессия продолжит со старого. Недопустимо.
- Автор в заголовке: `[Claude] что сделал`. Кратко, по-русски.
- Маленькие коммиты по одной логической задаче.
- Перед крупными правками убедись, что рабочая версия закоммичена (точка отката).
- В индексе не должно быть: `.env`, `build/`, `node_modules/`, `.claude/`,
  `.figma-tmp/`, чужих незакоммиченных файлов.
- Перед push — строка в `CHANGELOG.md`.
- **Pre-push hook** (один раз на машине): `git config core.hooksPath .githooks` —
  тогда `.githooks/pre-push` сам прогонит `npm run check` перед каждым push.
- Репозиторий — родительская папка (`…/AI project`), пути git с префиксом
  `Plugin for WP sites/`. Скрипты это учитывают (резолв от `git rev-parse --show-toplevel`).
