## Уроки QA (что понял после каждого отчёта)

**Отчёт 1 — S3 overlay шапки на hero (page-hero):**
- *Что понял:* «глобальная» находка оказалась НЕ глобальной. Из 3 hero-блоков
  перекрытие было только у `page-hero` (mobile padding-top 48px < шапки 84px).
  `region-hero` (local) уже резервировал `96px+r-40`, `hero-cover` (home) кладёт
  текст ниже картинки. Подтвердилось «не нашёл дефект = уже исправлено».
- *Урок:* НЕ чинить «на всех страницах» по формулировке клиента — проверять
  КАЖДЫЙ блок отдельно (у них разное устройство). И: «tablet»-жалоба может быть
  mobile-брейкпоинтом — 768px попадает под `max-width:782px`.
- *В тулинг:* нужен авто-чек «верх первой секции ≥ высоты фикс-шапки на каждом
  вьюпорте (390/768/1440)». Правило [[light-hero-top-padding]] существовало, но
  никто не проверял page-hero/mobile → дефект дожил до клиента. Проверка > правило.

**Отчёт 2 — S5/S6/S7 кнопки (батч):**
- *Что понял:* написал **детектор `scripts/qa-buttons.mjs`** — он сам, вслепую,
  по всем 10 страницам нашёл то, что я раньше пропускал: 7 кнопок без `href` +
  1 без hover. Это и есть прокачка скилла: мой QA теперь ловит S6/S7 сам. После
  фикса прогнал детектор → 0. S6 был из-за `buttonUrl:"#"` в сидерах (дефолт не
  переопределяли). S5 по высоте — ложная тревога (0), реальный дефект = ширина
  (кнопка во всю ширину на mobile) — детектор по высоте это пропустил.
- *Урок (в МОЙ QA):* (1) детектор кнопок теперь обязателен в гейте; (2) проверять
  не только высоту, но и **ширину** кнопок на mobile — добавить чек «кнопка ≈
  ширина вьюпорта = подозрение на растянутость»; (3) `href="#"` = «нет связи»,
  флагать как дефект.
