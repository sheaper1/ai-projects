# Visual QA — сверка живого сайта с Figma по числам

## Зачем так

Сравнивать дизайн и реализацию «на глаз» по скриншотам бесполезно: отступ в vw,
ширина колонки на 80px, вес шрифта 400 vs 300, отсутствие `<br>` в заголовке — на
ужатой картинке **физически не видны**. Поэтому QA здесь работает с **числами**:
живой DOM отдаёт точные значения (`getBoundingClientRect` + `getComputedStyle`)
даром и точно, Figma отдаёт эталонные числа через коннектор. Сверяем числа.

## Вход

Пользователь называет страницу («проверь Über mich», «сверь контакты»). Дальше
всё сам.

## Процедура

### 1. Резолв страницы → node-id (ВСЕГДА через карту)

Открой `AGENTS.md`, раздел **«Карта страниц (node-id desktop / mobile)»**. Найди
строку по имени или синониму (RU/EN). Возьми **оба** node-id: desktop и mobile.
Файл Figma: `p1HKLfoMcOwtVUD5rI9V3P`.

НЕ переискивай node-id заново и НЕ зови `get_metadata` по целой странице — это
~970k символов, не влезет в контекст. Ходи сразу по node-id фрейма из карты.

URL страницы на staging: `https://rosenberger.digirelation.dev/<slug>/` (slug
ищи в `scripts/pages/<...>.mjs` → `export const slug`).

### 2. Снять живые числа (мульти-ширина — это критично)

```
node scripts/qa-extract.mjs <url> --width 1440 --json .visual/<page>-1440.json
node scripts/qa-extract.mjs <url> --width 1920 --json .visual/<page>-1920.json
node scripts/qa-extract.mjs <url> --width 375  --json .visual/<page>-375.json
```

Скрипт сам скроллит страницу (иначе scroll-reveal оставляет секции пустыми).
**Две десктоп-ширины обязательны:** на 1440 реализация часто совпадает с Figma
случайно, а дефект vw-математики вылезает только на широком экране.

### 3. Снять эталон из Figma и СОХРАНИТЬ в файл

`get_metadata(fileKey, nodeId=<desktop из карты>)` — дерево с `x/y/width/height`
по каждому слою (имя слоя = текст). Сохрани XML-ответ через Write в
`.visual/<page>-figma.xml` (движок сравнения читает из файла). Mobile-фрейм —
отдельным вызовом по mobile node-id → `.visual/<page>-figma-m.xml`.

### 4. Запустить движок сравнения (он делает дифф, НЕ ты глазами)

```
node scripts/qa-compare.mjs --figma .visual/<page>-figma.xml \
     --live1440 .visual/<page>-1440.json --live1920 .visual/<page>-1920.json \
     --json .visual/<page>-diff.json
```

Движок джойнит элементы по тексту и выдаёт готовый детерминированный список:
vw-дрейф, потерянные переносы заголовков, расхождения ширины, выбросы веса,
**LIVE-ONLY** (лишние секции/иной текст), FIGMA-ONLY (потеряно), full-bleed
картинки, структурную дельту высоты. **Это и есть находки — твоя работа их
подтвердить и отсортировать, а НЕ пересравнивать на глаз и НЕ отменять.**

Mobile — повтори с `--figma .visual/<page>-figma-m.xml --live1440
.visual/<page>-375.json` (375 как «узкая» ширина).

### 5. Как читать вывод движка (важные нюансы)

- **vw-дрейф** — реальный дефект: hero-текст с фикс `x` в px вместо vw. НЕ
  отменяй его рассуждением «это сплит-hero / это gutter» — пользователь это
  считает дефектом. Сообщай как есть.
- **LIVE-ONLY** — интерпретируй по типу:
  - целый **контентный блок** (заголовок + абзацы), которого нет в Figma =
    **реальная лишняя секция** (или устаревший node-id в карте — предупреди);
  - тексты в зоне **band'а** (полноширинная картинка-оверлей) или **футера** =
    обычно неразвёрнутый Figma-компонент, НЕ дефект.
- **FIGMA-ONLY** не пусто = на лайве потеряли/заменили текст из дизайна.
- **Картинки**: «band'ов в дизайне: N>0» → растяжение штатное; «0 → ПОДОЗРЕНИЕ»
  → реальный дефект.

### 6. Чего движок НЕ видит — добери вручную

- **Лейбл кнопок/CTA** (напр. шапка «Kontakt» в дизайне vs «Termin vereinbaren»
  на лайве): `get_metadata` не отдаёт текст внутри компонента-кнопки. Сверь
  через `get_design_context(<нода шапки/кнопки>)` или глазами по скриншоту
  дизайна.
- **Точный вес vs дизайн**: движок ловит лишь внутренний выброс. Для «должно
  быть 300, а 400» возьми `get_design_context` нужной секции (там есть
  `font-weight`).
- **Поведение** (hover, drag карусели, отправка формы) — только живой браузер.

## Жёсткие правила-инварианты (часто без эталона)

- **Hero-инсет масштабируется как vw.** Полноэкранный hero по правилу проекта
  держит пропорцию Figma-фрейма через vw/%. Значит `leftPct` текста hero должен
  быть ОДИНАКОВ на 1440 и 1920. Если на 1440 = 3.3%, а на 1920 = 2.5% — инсет
  захардкожен в px вместо `vw`. Дефект.
- **Каждая кнопка обязана иметь `:hover`** (правило проекта) — статичный снимок
  не ловит, проверяй через браузерный канал если доступен.
- **Вес шрифта описаний = токен.** Если большинство body-абзацев weight 300, а
  один блок (обычно CTA-описание) weight 400 — это выброс, флагуй.
- **Заголовки повторяют переносы Figma через `<br>`.** Авто-перенос ≠ дизайн.

## ФАЗА 2 — парные скриншоты (обязательно для цвета/зазоров/италика/структуры)

Числа из Фазы 1 слепы к зазорам между элементами, цвету иконок/SVG,
`font-style`, теням, высотам и «выглядит криво». Это ловит только зрение —
парное сравнение рендера секции дизайна и той же секции лайва. Индустрия
(Applitools, Pixelay, UI Match) делает ровно overlay-сравнение; vision-only при
этом сам по себе промахивается (путает структуру), поэтому он ИДЁТ В ПАРЕ с
числами Фазы 1, а не вместо.

Для каждой подозрительной секции (и для блоков, где Фаза 1 что-то нашла):

1. **Рендер дизайна:** `get_screenshot(fileKey, nodeId=<нода секции из get_metadata>,
   maxDimension 1400)` → вернёт URL → `curl -o .visual/fig-<sec>.png "<url>"`.
2. **Та же секция лайва, изолированно:** `npm run shot -- <url> --sel "<css>"`
   (shot.mjs снимает один селектор). CSS-класс секции бери из разметки лайва
   (напр. `.problem-cards`, `.split-cta`, `.testimonials`).
3. **Read обе** и сравни ГЛАЗАМИ конкретно по списку, что числа не видят:
   - зазоры между элементами (иконка↔заголовок, заголовок↔текст);
   - цвет иконок/SVG/стрелок, фонов, рамок;
   - `font-style` (italic/`<em>`) — но сверяй С ДИЗАЙНОМ: если в Figma тоже
     италик, это НЕ дефект (гаси ложную тревогу);
   - тени, радиусы, наложения, выравнивание;
   - **структура**: есть ли на лайве блок, которого нет в дизайне (и наоборот).
4. Обе картинки изолированы и в ширине дизайна (1440) → выравнивание сходится
   само; не сравнивай полностраничные скрины целиком (главный провал DIY-тулов —
   рассинхрон кропа).

## ФАЗА 3 — поведение (живой браузер)

Карусель листается? Точки пагинации не `display:none`? Аккордеон раскрывается?
Ховеры на кнопках есть (инвариант проекта)? Форма шлёт? Это статикой не взять —
проверяй через браузерный канал, если доступен.

> Реалистичная планка: Фаза 1+2+3 ловят ~80–90%, хвост — финальный проход
> глазами владельца. Цель не оракул, а всё меньше пропусков на каждом прогоне.

## Формат отчёта — ПО БЛОКАМ

Движок уже группирует находки по секциям («Дефекты по блокам сверху вниз»).
Отчёт строй так же — по блокам, чтобы было видно «блок → его проблемы», а не
свалку по типам:

```
# Visual QA: <страница>  (staging ↔ Figma <node-id>)

## Блок «Hero»
- 🔴 vw-отступ: hero-текст x=48px фикс → должен быть vw (Figma 3.3% → 1920 2.5%)
- 🔴 вес: «…» weight 400 вместо 300

## Блок «Was Eigentümer…»
- 🔴 перенос: заголовок Figma 3 стр → лайв 2 (нет <br>)
- 🔴 ширина: колонка Figma 504 → лайв 646 (Δ+142)

## Структура / лишнее
- лишняя секция на лайве: «…» (или: реальный контент vs плейсхолдер в Figma)
- порядок блоков, full-bleed картинки, Δ высоты

## ✅ Блоки без расхождений: <перечисли>
```

Каждый дефект — с числами «эталон vs факт», без «на глаз». Не выдавай
предположение за факт: если эталон неоднозначен (текст в дизайне 2×, фрейм мог
устареть, плейсхолдер vs реальный контент) — помечай «нужна проверка».

## Бандл (скрипты в репозитории проекта)

- `scripts/qa-extract.mjs <url> --width N --json out` — снимает живые числа с DOM.
- `scripts/qa-compare.mjs --figma <xml> --live1440 <json> [--live1920 <json>]` —
  движок сравнения: джойн по тексту + детерминированный дифф. Вся логика поиска
  дефектов здесь, чтобы агент не сравнивал на глаз и не отменял находки.
