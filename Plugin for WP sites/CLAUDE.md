# Библиотека блоков WordPress + конвейер сборки сайтов

Система для веб-агентства: эталонная библиотека динамических блоков Gutenberg, из
которой собираются сайты под клиентов. Контент клиент редактирует сам в Gutenberg,
макет защищён блокировкой.

## Архитектурные решения (зафиксированы)

- **Модель блока:** `InnerBlocks` + `templateLock: "all"`. Клиент правит вложенные
  ядровые блоки нативным Gutenberg, но не может добавлять/удалять/переставлять
  структуру.
- **Сборка JS:** `@wordpress/scripts` (JSX, `npm run build`). Требуется Node/npm.
- **Проект = блочная тема.** Каждый `/projects/<сайт>` — block theme со своим
  `theme.json` и папкой `blocks/`. Блоки регистрируются из `functions.php` темы.

## Структура папок

```
/library/                 — ЭТАЛОННАЯ библиотека (read-only при работе над проектами)
  blocks-registry.md      — каталог всех блоков
  /blocks/<slug>/         — самодостаточная папка блока (копируется целиком)
/projects/<имя-сайта>/    — конкретный проект (блочная тема: theme.json + blocks/)
theme.json                — базовый эталон глобальных стилей (шаблон для проектов)
package.json              — сборка библиотеки
```

## Рабочий цикл

1. **Новый блок** создаётся СНАЧАЛА в `/library/blocks/<slug>/`, доводится до конца.
2. Готовый блок **копируется** из `/library` в проект (вся папка блока).
3. Правится **всегда только копия в проекте**. Эталон в `/library` НЕ трогаем.
4. Перед созданием нового блока **сверься с `blocks-registry.md`**: если похожий
   есть — предложи скопировать его; если нет — создавай новый и **допиши в реестр**.

## Жёсткие правила блоков

- **Только динамические блоки** (`render.php`). Никакого статического HTML в `save`.
- **Стили только через CSS-переменные и `theme.json`.** Запрещены захардкоженные
  цвета, шрифты, размеры шрифтов. Используй `var(--wp--preset--…)` и пресеты.
- **Глобальные стили — в `theme.json` проекта.** Поменял значение токена →
  поменялось везде.
- **Клиент удобно редактирует текст и картинки** в Gutenberg (ядровые блоки внутри
  `InnerBlocks`).
- **Макет защищён** через `templateLock: "all"`.

## Структура одного блока

```
/library/blocks/<slug>/
  block.json          — apiVersion 3; ссылки на build/ и render.php
  render.php          — серверный рендер: обёртка + $content
  src/
    index.js          — регистрация блока
    edit.js           — InnerBlocks с шаблоном и templateLock
    save.js           — возвращает <InnerBlocks.Content />
    style.scss        — фронт + редактор, ТОЛЬКО CSS-переменные
  build/              — генерируется `npm run build`; в git НЕ коммитим (.gitignore)
```

## Сборка и поставка

```
npm install
npm run build          # собрать все блоки
```

**Блоки поставляются как плагин `library-blocks`** (`library/library-blocks.php`
регистрирует все блоки из `blocks/*` штатно — editorScript/style резолвятся через
`plugins_url`, без хаков). **Проект = блочная тема** (`projects/<name>/` с `theme.json`).

**Деплой на staging (без SFTP):** `node scripts/deploy-stack.mjs` — собирает плагин
+ тему, заливает файлы и активирует их через одноразовый сниппет Code Snippets,
который сам себя обезвреживает. Подробности доступов — в `.env` (gitignore).

> `build/` в git не хранится — нужен `npm run build` перед деплоем.
> ВАЖНО: в этой версии Code Snippets REST DELETE сломан (500); наши служебные
> сниппеты не удаляются, а обезвреживаются (деактивация + пустой код).

## Работа с Git

Над проектом **по очереди** работают два ИИ — Claude и Codex (никогда
одновременно). Чтобы не терять работу друг друга:

- **Начало работы → всегда `git pull`** (стянуть свежую версию).
- **Конец работы → всегда `git push`** (залить изменения). Забыл push —
  второй ИИ продолжит со старой версии. Это недопустимо.

**Коммиты:**

- Указывай автора в заголовке. Формат: `[Claude] что сделал` (Codex — `[Codex] …`).
- Маленькие коммиты, по одной логической задаче.
- Описание кратко, на русском.

**Версии и безопасность:**

- Перед крупными изменениями убедись, что рабочая версия закоммичена — чтобы
  было куда откатиться (`git revert` для запушенного, `git reset` для локального).
- **Не коммить лишнее:** `node_modules/`, ключи, пароли, `.env`, собранные файлы
  (`build/`). Всё это уже в `.gitignore` — поддерживай его в актуальном виде.
- Перед push проверь, что в индексе нет секретов и артефактов.
