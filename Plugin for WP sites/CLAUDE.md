# Библиотека блоков WordPress + конвейер сборки сайтов

Система для веб-агентства: эталонная библиотека динамических блоков Gutenberg, из
которой собираются сайты под клиентов. Контент клиент редактирует сам в Gutenberg,
макет защищён блокировкой.

## Архитектурные решения (зафиксированы)

- **Модель блока:** `InnerBlocks` + `templateLock: "all"`. Клиент правит вложенные
  ядровые блоки нативным Gutenberg, но не может добавлять/удалять/переставлять
  структуру.
- **Сборка JS:** `@wordpress/scripts` (JSX, `npm run build`). Требуется Node/npm.
- **Проект = блочная тема.** Каждый `/projects/<сайт>` — block theme со своим
  `theme.json` и папкой `blocks/`. Блоки регистрируются из `functions.php` темы.

## Структура папок

```
/library/                 — ЭТАЛОННАЯ библиотека (read-only при работе над проектами)
  blocks-registry.md      — каталог всех блоков
  /blocks/<slug>/         — самодостаточная папка блока (копируется целиком)
/projects/<имя-сайта>/    — конкретный проект, состоит из двух частей:
  /theme/                 — блочная тема (ВИД): theme.json, шаблоны, blocks/
  /plugin/<name>-core/    — плагин проекта (ДАННЫЕ/ЛОГИКА): настройки, CPT, формы
theme.json                — базовый эталон глобальных стилей (шаблон для проектов)
package.json              — сборка/деплой
```

> **Разделение «вид ↔ данные»:** блоки/оформление — в теме; CPT, кастомные поля,
> настройки сайта (контакты), формы — в плагине проекта (чтобы данные пережили
> смену темы). Плагин проекта **per-project**, не общий на всех клиентов.
>
> **Клиент редактирует один таб** «Настройки сайта» (плагин проекта), а шапка/подвал
> и блоки тянут эти данные через **Block Bindings** (`rosenberger/setting`). В Site
> Editor клиент не заходит; меню — нативное (Внешний вид → Меню).

## Рабочий цикл

1. **Новый блок** создаётся СНАЧАЛА в `/library/blocks/<slug>/`, доводится до конца.
2. Готовый блок **копируется** из `/library` в проект (вся папка блока).
3. Правится **всегда только копия в проекте**. Эталон в `/library` НЕ трогаем.
4. Перед созданием нового блока **сверься с `blocks-registry.md`**: если похожий
   есть — предложи скопировать его; если нет — создавай новый и **допиши в реестр**.

## Жёсткие правила блоков

- **Только динамические блоки** (`render.php`). Никакого статического HTML в `save`.
- **Стили только через CSS-переменные и `theme.json`.** Запрещены захардкоженные
  цвета, шрифты, размеры шрифтов. Используй `var(--wp--preset--…)` и пресеты.
- **Глобальные стили — в `theme.json` проекта.** Поменял значение токена →
  поменялось везде.
- **Клиент удобно редактирует текст и картинки** в Gutenberg (ядровые блоки внутри
  `InnerBlocks`).
- **Макет защищён** через `templateLock: "all"`.

## Глобальная дизайн-система проекта

Для проекта Rosenberger единственный источник дизайн-токенов —
`projects/rosenberger/theme/theme.json`. Новый блок сначала собирается из этих
пресетов; локальная CSS-переменная допустима только как алиас на глобальный токен.

- Контейнеры: `contentSize` для текста, `wideSize` для секций и сеток.
- Боковой отступ: `var(--wp--custom--layout--gutter)`.
- Вертикальный отступ секции: `var(--wp--custom--layout--section-space)`.
- Цвета: семантические пресеты `base`, `contrast`, `primary`, `accent`, `surface`,
  `surface-muted`, `muted`, `border`, `placeholder`, `on-dark`.
- Шрифты: `var(--wp--preset--font-family--body)` и
  `var(--wp--preset--font-family--heading)`.
- Размеры текста и интервалы: только проектные пресеты
  `--wp--preset--font-size--r-*` и `--wp--preset--spacing--r-*`. Префикс `r-`
  обязателен: стандартные slug'и Gutenberg могут быть переопределены сохранёнными
  Global Styles в базе WordPress.
- Радиусы: `--wp--custom--radius--small|medium|large|pill`.

Не создавай в блоке собственную палитру, шкалу размеров или контейнер. Если
дизайну нужен новый повторяемый параметр, сначала добавь семантический токен в
`theme.json`, затем используй его в блоке.

## Структура одного блока

```
/library/blocks/<slug>/
  block.json          — apiVersion 3; ссылки на build/ и render.php
  render.php          — серверный рендер: обёртка + $content
  src/
    index.js          — регистрация блока
    edit.js           — InnerBlocks с шаблоном и templateLock
    save.js           — возвращает <InnerBlocks.Content />
    style.scss        — фронт + редактор, ТОЛЬКО CSS-переменные
  build/              — генерируется `npm run build`; в git НЕ коммитим (.gitignore)
```

## Сборка и поставка

**Модель — «копия в проект» (НЕ общий плагин).** `/library` — эталоны, их копируем.
Каждый проект (`projects/<name>/`) — **самодостаточная блочная тема**: блоки лежат
внутри неё в `blocks/<slug>/`, тема их регистрирует из `functions.php`, плюс свой
`theme.json` (бренд). Блоки адаптируются под дизайн конкретного проекта здесь.

Почему так: дизайны у клиентов разные, блок всё равно адаптируется под каждый
проект. Улучшения эталона достаются будущим проектам; действующие — это форки, и
централизованно их не ломаем. «Новые фишки клиенту» приносим управляемо, по проекту.

```
npm install
npm run build              # собрать блоки активного проекта (projects/rosenberger)
npm run build:library      # собрать эталоны (для проверки)
```

**Деплой на staging (без SFTP):** `node scripts/deploy-stack.mjs` — собирает тему,
заливает файлы и активирует её через одноразовый сниппет Code Snippets, который сам
себя обезвреживает. Доступы — в `.env` (gitignore).

> `build/` в git не хранится — нужен `npm run build` перед деплоем.
> ВАЖНО: в этой версии Code Snippets REST DELETE сломан (500); служебные сниппеты
> не удаляются, а обезвреживаются (деактивация + пустой код).

## Работа с Git

Над проектом **по очереди** работают два ИИ — Claude и Codex (никогда
одновременно). Чтобы не терять работу друг друга:

- **Начало работы → всегда `git pull`** (стянуть свежую версию).
- **Конец работы → всегда `git push`** (залить изменения). Забыл push —
  второй ИИ продолжит со старой версии. Это недопустимо.

**Коммиты:**

- Указывай автора в заголовке. Формат: `[Claude] что сделал` (Codex — `[Codex] …`).
- Маленькие коммиты, по одной логической задаче.
- Описание кратко, на русском.

**Версии и безопасность:**

- Перед крупными изменениями убедись, что рабочая версия закоммичена — чтобы
  было куда откатиться (`git revert` для запушенного, `git reset` для локального).
- **Не коммить лишнее:** `node_modules/`, ключи, пароли, `.env`, собранные файлы
  (`build/`). Всё это уже в `.gitignore` — поддерживай его в актуальном виде.
- Перед push проверь, что в индексе нет секретов и артефактов.

## Обязательное завершение каждой задачи

Для любой **завершённой задачи, которая меняет код, тему, плагин или контент
проекта**, исполнитель без дополнительного напоминания проходит весь цикл:

1. Перед работой сверяет локальную ветку с удалённой (`git pull`). Если уже есть
   незакоммиченные изменения, сначала определяет их владельца и не перетирает их.
2. Выполняет релевантную проверку: сборку, PHP syntax check, lint или прямую
   проверку результата.
3. Выгружает актуальную тему и project-core на WordPress staging командой
   `node scripts/deploy-stack.mjs`.
4. Проверяет, что тема и плагин активны и staging отвечает без ошибки. Для
   визуальных изменений дополнительно проверяет изменённую страницу в браузере,
   когда браузерный канал доступен.
5. Коммитит только связанные с задачей файлы. Формат сообщения:
   `[Claude] ...` или `[Codex] ...`, кратко и по-русски.
6. Выполняет `git push` и убеждается, что удалённая ветка приняла коммит.

Если сборка, деплой или проверка не прошли, не выдавай задачу за завершённую:
исправь проблему либо явно сообщи блокер. Не добавляй в коммит `.env`, `build/`,
`.figma-tmp/`, `node_modules/` и чужие незакоммиченные файлы.

Для анализа, ответов на вопросы и других задач без изменения проекта деплой и
пустой коммит не требуются.

## Работа с Figma

- Источник дизайна Rosenberger — файл Figma `3AzuInZ4YD95cLiQgiD24W`.
- Перед созданием блока сначала читай конкретный frame/node и используй точные
  значения макета; не подбирай цвета, размеры и интервалы визуально, если Figma
  отдаёт их программно.
- Полный ответ Figma может находиться во вложенном
  `structuredContent.content`, даже если верхний `content` содержит только
  `Action completed`.
- Повторяемое значение сначала оформляется глобальным токеном в `theme.json`, и
  только затем используется блоком. Локальный токен допустим как алиас на
  глобальный, но не как параллельная дизайн-система.
- Полноэкранные Hero на desktop сохраняют пропорцию исходного Figma-frame через
  `vw`/проценты; координаты контента масштабируются вместе с секцией. Фон всегда
  `cover`, а `object-position` проверяется минимум на 1440 и 1920 px. На mobile
  разрешён отдельный `100svh` layout.
- Иконки хранятся как отдельные SVG в WordPress Media Library. Блок хранит media
  ID/URL и предоставляет `MediaUpload`; inline SVG, icon-font и привязка иконки к
  пути темы запрещены. Исключение — логотип/служебный asset самой темы.
